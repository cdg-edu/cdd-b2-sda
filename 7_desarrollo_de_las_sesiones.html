7. Desarrollo de las sesiones

TAREA 1: Exploración de APIs abiertas de la Unión Europea

Duración: 1 sesión (2 horas)

Formato: Trabajo por grupos cooperativos

Descripción de la tarea

En esta primera tarea, el alumnado explorará diversas fuentes de datos abiertas relacionadas con el medioambiente, en particular calidad del aire y del agua, proporcionadas por instituciones como la Agencia Europea del Medio Ambiente (EEA) o el European Data Portal. El objetivo es seleccionar una API, comprender su estructura y preparar una infografía explicativa.

Texto explicativo (introducción al contexto)

Los datos abiertos son recursos públicos que pueden ser utilizados libremente para desarrollar soluciones digitales que ayuden a la sociedad. En esta tarea, usarás estos datos para estudiar cómo se mide y publica la calidad del aire y del agua en Europa, y cómo puedes utilizarlos en tus programas.

Vídeo

  • ¿Qué es una API?
    https://youtu.be/s7wmiS2mSXY
  • Datos abiertos y sostenibilidad (EEA)
    https://youtu.be/dZxE2E0Ekks

Material de apoyo

  • Recursos para la búsqueda
    • European Data Portal
    • EEA - European Environment Agency
    • OpenAQ API
    • Datos.gob.es
  • Preguntas de repaso
    • ¿Qué es una API RESTful?
    • ¿Qué significa JSON?
    • ¿Qué tipo de datos son relevantes para la calidad del aire?
    • ¿Por qué es importante filtrar por parámetros y ciudades?

Diccionario de términos clave

  • API: Interfaz que permite acceder a funcionalidades o datos desde una app externa.
  • JSON: Formato estándar para el intercambio de datos (JavaScript Object Notation).
  • Endpoint: Dirección URL que permite acceder a una parte específica de una API.
  • Parámetro: Información añadida a una consulta API para personalizar los datos obtenidos.
  • Partículas PM: Contaminantes del aire, como PM10 o PM2.5, que afectan a la salud.
  • GET: Tipo de solicitud HTTP que se usa para obtener datos de una API.

Actividades del alumnado

  1. Buscar APIs de datos ambientales
    Usando el European Data Portal o EEA, identificar al menos una API que proporcione datos de calidad del aire o del agua.
  2. Analizar su estructura
    ¿Qué formato tiene? ¿Qué parámetros admite?
    ¿Qué devuelve? ¿Hay ejemplos?
  3. Crear una infografía digital
    Resumen con: nombre de la API, ejemplo de endpoint, parámetros importantes, formato de salida.
  4. Subir la infografía a la plataforma del centro
    Puede ser OneNote, Teams, Aula Virtual, Moodle o GitHub Classroom si se usa como portfolio.

Herramientas y soporte digital

  • Búsqueda y edición: Navegador web, Google Docs, Canva, PowerPoint Online.
  • Presentación: OneNote, Teams, Moodle o Aules.
  • Organización: Trello o Planner (actividad de bonus: crear tablero del proyecto).
  • Evaluación inicial: Cuestionario de repaso con Forms o Kahoot.

TAREA 2: Diseño del flujo del programa y modelo de datos

Duración: 1 sesión (2 horas)

Formato: Trabajo por grupos cooperativos

Objetivo de la tarea

El alumnado diseñará el funcionamiento lógico de su programa y las estructuras de datos que se utilizarán para procesar la información obtenida desde una API abierta relacionada con el medioambiente.

Contenidos y formatos de trabajo

  1. Producciones del alumnado (por grupo):
    • Diagrama de flujo del funcionamiento del programa (Draw.io, Lucidchart, Miro o papel y foto).
    • Especificación de estructura de datos a usar (en pseudocódigo, JSON, Java u otro formato).
    • Presentación oral breve o revisión entre grupos para validar la lógica del diseño.
    • Vídeo tutorial breve (1-2 minutos) mostrando cómo organizarán su trabajo en una herramienta colaborativa (por ejemplo, Trello o Miro).
  2. Formatos utilizados:
    • Diagrama visual (imagen JPG o PNG)
    • Documento con pseudocódigo o esquema de estructuras
    • Vídeo explicativo (grabado con Loom, Clipchamp o móvil)
    • Plantilla de esquema (proporcionada en PDF o editable)

Material de apoyo / ampliación

  • Guía de pseudocódigo básica en PDF
  • Vídeo: Cómo hacer un diagrama de flujo paso a paso
  • Tutorial básico: ¿Qué es Trello y cómo organizar tu proyecto?
  • Plantilla editable de estructura de datos en JSON: 
    {
    "ciudad": "Barcelona",
    "fecha": "2024-10-05",
    "parametro": "pm10",
    "valor": 43.5,
    "unidad": "µg/m³"
    }

Diccionario de términos clave

  • API: Interfaz que permite acceder a datos o funcionalidades externas
  • JSON: Formato de texto ligero para representar estructuras de datos
  • Pseudocódigo: Representación informal de un algoritmo
  • Diagrama de flujo
  • Representación gráfica del proceso lógico del programa
  • Objeto: Estructura de datos que agrupa información en forma de atributos
  • Trello: Herramienta de gestión de proyectos en forma de tablero visual

Soporte digital de trabajo

Los alumnos trabajarán y entregarán sus producciones en:

  • Teams (canal del grupo): publicación de diagramas y pseudocódigo.
  • Aula Virtual / Moodle / Classroom: entrega del documento y vídeo en tarea asignada.
  • OneNote (cuaderno de grupo): recopilación de recursos usados y anotaciones del proceso.

TAREA 3: Implementación de parseo de datos y modularización del código

Duración: 2 sesiones (4 horas)

Formato: Trabajo por equipos en entorno colaborativo

Objetivo de la tarea
El alumnado implementará la conexión con una API, organizará el código en funciones o clases y realizará el tratamiento básico de los datos. Además, añadirá filtros útiles (por ciudad, contaminante, etc.) y validaciones mínimas sobre los datos obtenidos.

Contenidos y formatos de trabajo

  1. Producciones del alumnado (por grupo):
    • Código fuente organizado de forma modular (funciones, clases).
    • Conexión funcional a una API con recuperación y parseo de datos.
    • Aplicación de filtros básicos (ubicación, tipo de parámetro).
    • Validación de campos clave (ej. existencia de fecha, valor, unidad…).
    • Pruebas básicas de funcionamiento (entradas válidas y no válidas).
    • Registro de avances en Trello/Miro u otra herramienta elegida.
  2. Entregables:
    • Carpeta de proyecto en Java/Python/JavaScript con el código funcional.
    • Capturas de pantalla con la salida de pruebas.
    • Informe breve de validación y testeo (puede incluir errores comunes y cómo se resolvieron).

Material de apoyo y ampliación

  • Ejemplo comentado de código en Java para conexión a API REST y parseo JSON (disponible en el Aula Virtual).
  • Vídeo: Cómo estructurar tu código en funciones y módulos
  • Tutorial: Cómo usar Postman para probar APIs
  • Cheatsheet de validaciones básicas:
    • Verificar que la respuesta no está vacía
    • Comprobar que campos como "value", "date" o "unit" están presentes
    • Manejar errores de conexión (try-catch, bloques try/except, etc.)

Diccionario de términos clave

  • Parsear: Procesar y transformar datos desde un formato externo (ej. JSON) a estructuras internas del programa
  • Función / Método: Bloque de código reutilizable con una tarea concreta
  • Filtro: Lógica que selecciona datos según un criterio definido (ej. ciudad = “Madrid”)
  • Validación: Comprobación de que los datos recibidos cumplen con condiciones mínimas para su uso
  • Modularización: Organización del código en componentes o funciones que facilitan la reutilización y el mantenimiento

Soporte digital y herramientas utilizadas

  • IDE (Visual Studio Code, IntelliJ, Eclipse o similar): para programar y probar código.
  • Postman o navegador: para pruebas de la API.
  • GitHub / Teams / Moodle: entrega del código y evidencias del trabajo.
  • Trello / Miro: seguimiento de tareas y comunicación en el equipo.

Preguntas de reflexión (al finalizar la tarea)

  • ¿Qué parte del parseo fue más difícil? ¿Cómo lo resolviste?
  • ¿Cómo has dividido tu código y por qué?
  • ¿Qué validaciones crees que son imprescindibles cuando trabajas con datos externos?
  • ¿Qué ocurre si la API devuelve un error o no hay conexión?

TAREA 4: Visualización simple de resultados (texto o HTML)

Duración: 2 sesiones (4 horas)

Formato: Desarrollo técnico individual o por parejas, con presentación final

Objetivo de la tarea
Los estudiantes deben implementar una visualización comprensible y legible de los datos extraídos y tratados. La presentación puede hacerse en texto formateado por consola o, preferentemente, en una página HTML generada desde código, resaltando los datos más relevantes (por ejemplo, contaminantes elevados).

Contenidos y producciones esperadas

  1. Producciones del alumnado
    • Tabla con los datos clave de la API (fecha, ubicación, tipo y valor del contaminante).
    • Visualización generada por consola o en HTML básico.
    • Resaltado de valores críticos con colores o mensajes.
    • Archivo .html que puede abrirse en navegador o visualización directa desde consola.
    • Comentarios explicativos en el código fuente.
  2. Entregables
    • Código fuente actualizado.
    • Archivo .html generado o capturas de la visualización en consola.
    • Evidencias del resaltado visual o textual de valores críticos (por ejemplo, si una medida supera un umbral recomendado).
    • Registro en Trello/Miro con avances y dificultades encontradas.

Material de apoyo y ampliación

  • Ejemplo de tabla HTML con estilos básicos (<table>, <tr>, <th>, <td>) y coloración condicional con CSS.
  • Plantilla base para mostrar datos en consola en formato alineado.
  • Tutorial rápido: Generar HTML dinámicamente con Java o Python
  • Recurso: Umbrales de calidad del aire de la UE (PDF)

Diccionario de términos clave

  • HTML: Lenguaje de marcado para construir páginas web
  • Visualización: Representación gráfica o estructurada de los datos
  • Estilo condicional: Técnica para aplicar colores o formato según los valores
  • Interfaz: Punto de interacción entre el usuario y el sistema
  • Consola: Ventana de texto que muestra información y permite interacción con programas

Soporte digital y herramientas utilizadas

  • IDE (Java, Python, JavaScript).
  • Navegador Web para visualizar el archivo HTML generado.
  • Trello / Miro para la planificación.
  • Aula Virtual / GitHub para entrega de resultados y código.

Preguntas de reflexión

  • ¿Qué formato de salida (HTML o consola) te ha parecido más claro para el usuario?
  • ¿Cómo has destacado los datos críticos? ¿Por qué crees que es importante?
  • ¿Serías capaz de adaptar tu visualización para mostrar más tipos de datos?
  • ¿Tu código puede adaptarse a otros conjuntos de datos fácilmente?

TAREA 5: Documentación y presentación del proyecto final

Duración: 2 sesiones (4 horas)

Formato: Trabajo por equipos – Exposición y reflexión final

Objetivo de la tarea
El alumnado consolidará todo el trabajo realizado documentando el proyecto, comentando su código, y preparando una presentación clara de su aplicación. La tarea culmina con la exposición de los proyectos ante el grupo-clase y la entrega del material como evidencia de aprendizaje.

Contenidos y producciones esperadas

  1. Producciones del alumnado
    • Memoria del proyecto que incluya:
      • Introducción y objetivos.
      • Fuentes de datos utilizadas.
      • Diseño del flujo del programa y modelo de datos.
      • Código fuente con comentarios.
      • Capturas o resultados del programa.
      • Reflexión sobre los retos y logros del equipo.
    • Presentación oral (5–10 min por grupo).
    • Entrega del código en repositorio (GitHub o similar).
  2. Entregables
    • Documento de memoria (Word, PDF o similar).
    • Diapositivas o presentación visual.
    • Código fuente completo, comentado y organizado.
    • Registro final del tablero de trabajo colaborativo (Trello, Miro, etc.).
    • Enlace al repositorio público del proyecto (opcionalmente con README).

Material de apoyo y ampliación

  • Plantilla de memoria del proyecto en Word / Google Docs.
  • Guía rápida: Cómo comentar correctamente el código.
  • Guía para crear una presentación efectiva.
  • Ejemplos de proyectos similares (con fines ilustrativos).
  • Lista de verificación final para repasar antes de la entrega.

Diccionario de términos clave

  • Memoria del proyecto: Documento que recoge todo el trabajo realizado en el proyecto.
  • README: Archivo de introducción y documentación breve incluido en un repositorio.
  • Repositorio: Espacio online donde se guarda y versiona el código (GitHub, GitLab, etc.).ç
  • Presentación oral: Exposición verbal estructurada para comunicar el proyecto.
  • Reflexión: Análisis personal o grupal sobre lo aprendido y lo vivido durante el proyecto.

Soporte digital y herramientas utilizadas

  • Word / Google Docs / Office365 para la memoria escrita.
  • PowerPoint / Google Slides / Canva para la presentación.
  • GitHub para compartir el código y documentación.
  • Trello / Miro / Jamboard como evidencia del trabajo colaborativo.
  • Aula Virtual / OneDrive / Teams para la entrega y retroalimentación.

Actividades del alumnado

  • Recopilación y síntesis del trabajo realizado.
  • Revisión y comentarios del código fuente.
  • Redacción colaborativa de la memoria.
  • Preparación de la presentación visual.
  • Simulación o ensayo de la exposición oral.
  • Entrega del proyecto final a través del aula virtual o GitHub.

Preguntas de reflexión

  • ¿Qué parte del proyecto te ha parecido más difícil y por qué?
  • ¿Qué has aprendido sobre programación, datos abiertos o sostenibilidad?
  • ¿Cómo ha influido el trabajo en equipo en tu resultado final?
  • ¿Qué mejorarías si tuvieras que repetir el proyecto?
  • ¿Cómo aplicarías lo aprendido en un nuevo contexto real?
